# GoodClean 项目开发规则

## Git 工作流

### 分支策略
- `master`: 主分支，只接受合并，不直接提交代码
- `dev`: 开发分支，所有新功能和修复都在 dev 上开发

### 合并规则（重要！）
- **禁止自动合并到 master**
- 必须等用户亲自体验 dev 分支的功能后，确认无问题才能合并
- 合并前必须询问用户："是否确认合并到 master？"
- 用户同意后才执行 `git checkout master && git merge dev`

### 提交规范
- `feat:` 新功能
- `fix:` 修复 Bug
- `chore:` 杂项（清理文件、更新配置等）
- `docs:` 文档更新
- `refactor:` 代码重构

### 开发流程
1. 开发新功能前先切到 dev 分支
2. 在 dev 上开发、测试、提交
3. 通知用户在 dev 分支上体验
4. 用户确认后，询问是否合并到 master
5. 合并后推送到远程

## 项目规则

### 代码质量
- 修复 Bug 时，必须全面检查相关代码，避免修复一个 Bug 引入新的问题
- 耗时操作必须提供进度反馈（notify 或进度指示）
- 所有 UI 组件必须有明确的 CSS 样式定义

### 依赖管理
- 核心依赖：textual, rich, send2trash
- 不引入不必要的依赖

### 缓存策略
- 扫描结果缓存到 ~/.goodclean/cache/
- 缓存有效期 24 小时
- 增量扫描时自动复用未变化的目录数据
